# Benchmark Run Manager

Interactive CLI tool built with oclif for configuring and tracking benchmark runs.

## Installation

```bash
cd Scenarios/runner
npm install
```

## Usage

```bash
npm run setup
```

Or directly with node:

```bash
node setup-run.js
```

## Features

- **Interactive Prompts** - Guided configuration with sensible defaults
- **Organized Output** - Auto-creates run directories with consistent structure
- **Podman Integration** (REST only) - Builds container images with your settings
- **ECR Push** (REST only) - Automatically pushes images to AWS ECR
- **AWS Bootstrap** (REST only) - Creates VPC, security groups, RDS, ECS cluster, and ALB
- **Task Deployment** (REST only) - Deploys task definitions and restarts services
- **Configuration Files** - Saves `run_config.json` for reproducibility
- **ECS Task Definition** (REST only) - Generates ready-to-deploy task definitions
- **Virtual Threads** - Dynamic runtime configuration via environment variables
- **Documentation** - Generates `README.md` template for results
- **Colored Output** - Better readability with chalk
- **Verbose Mode** - Use `--verbose` flag for detailed output

## What It Does

1. **Prompts for Configuration**
   - Scenario type (REST or Kafka)
   - Load test scenario (Read Heavy, Balanced, Write Heavy, Ramp Up, Burst Spike, Cold Start, Warm Start)
   - Java version (17, 21, 25, or **GraalVM Native Image**)
   - JVM options (N/A for GraalVM native)
   - CPU and Memory settings (auto-adjusted for GraalVM)
   - Virtual Threads enabled/disabled
   - Database pool size

2. **For REST Scenarios:**
   - Reads the appropriate Dockerfile template for the Java version
   - Injects JVM options into the Dockerfile
   - Builds a Podman image tagged with run identifier
   - Reads ECS task definition template
   - Updates task definition with image name, CPU, memory, and DB pool size
   - Saves both Dockerfile and task definition to run directory

3. **Creates Run Directory Structure**
   ```
   runs/Scenarios/{REST|Kafka}/{scenario_name}/{java-version-folder}/{date}/{run_number}/
   ├── run_config.json              # Configuration saved as JSON
   ├── README.md                    # Markdown template for results
   ├── Dockerfile                   # Modified Dockerfile (REST only)
   └── task-definition.json         # ECS task definition (REST only)
   ```
   
   **Note:** For Java versions, the folder uses `java17`, `java21`, `java25`, or `graalvm` (for GraalVM native image).

4. **Generates Configuration Files**
   - Stores all settings in `run_config.json` for reproducibility
   - Timestamp and run ID recorded for each run
   - Task definition ready to deploy immediately

## AWS Bootstrap Feature

When configuring a REST scenario run, you'll be asked: **"Do you want to bootstrap the AWS infrastructure?"**

If yes, the CLI will:

1. **Prompt for AWS Components** to set up:
   - ECS Fargate Cluster (default: enabled)
   - RDS PostgreSQL Database (default: enabled)
   - Application Load Balancer (default: enabled)
   - Prometheus Monitoring (default: disabled)

2. **Collect AWS Configuration**:
   - AWS Region (default: us-east-1)
   - RDS master password

3. **Create Infrastructure** using AWS SDK:
   - Discovers default VPC and available subnets
   - Creates security groups for ALB, ECS, and RDS
   - Sets up RDS PostgreSQL database
   - Creates ECS cluster with Container Insights
   - Creates Application Load Balancer
   - Configures Prometheus (if selected)

### Implementation

The AWS bootstrap functionality is implemented in a separate module:

- **`aws-bootstrap.js`** - Core AWS infrastructure management using AWS SDK
- **`AWS_BOOTSTRAP_README.md`** - Detailed documentation for the AWS bootstrap module

This separation keeps the main `setup-run.js` file focused on CLI configuration while offloading AWS infrastructure concerns to a dedicated module.

## Next Steps After Running

### For REST Scenarios

1. **Podman Image is Built**
   - Podman image tagged: `benchmark-app:{runId}`
   - Image includes your JVM options
   - Ready for deployment

2. **Deploy to ECS**
   - Use the generated `task-definition.json`
   - Register it in AWS:
     ```bash
     aws ecs register-task-definition --cli-input-json file://task-definition.json
     ```
   - Update the service to use new task definition
   - Note: Push image to ECR or your container registry as needed

3. **Run Load Test**
   - Once deployed, run the k6 scenario
   - Capture results and update README.md

### For Kafka Scenarios

1. **Manual Setup**
   - Create Docker image separately using instructions in Kafka scenario folder
   - Configure cluster manually

2. **Run Load Test**
   - Execute appropriate k6 scenario
   - Capture results and update README.md

### All Scenarios

3. **Document Results**
   - Update the generated `README.md` with test results
   - Include metrics from:
     - k6 output (throughput, latency, error rate)
     - Container Insights (CPU, memory)
     - RDS CloudWatch (connections, latency)
     - Prometheus (JVM metrics, GC)

4. **Analysis**
   - Compare runs with different configurations
   - Use `run_config.json` to ensure consistency
   - Track improvements across Java versions

## Deploying Tasks to ECS

Once infrastructure is bootstrapped and you have run configurations, deploy them to ECS:

```bash
npm run deploy-task
```

This command:

1. **Selects a Run Configuration** - Lists all available run configs (most recent first)
2. **Prompts for AWS Settings**:
   - AWS Region (default: `us-east-1`)
   - ECS Cluster Name (default: `java-bench-cluster`)
   - ECS Service Name (default: `java-bench-service`)
   - Force new deployment flag
3. **Registers Task Definition** - Creates a new revision of the task in ECS
4. **Updates Service** - Points the service to the new task definition
5. **Monitors Deployment** - Shows running task count and status

## GraalVM Native Image Option

When you select **GraalVM** as the Java version, the setup process:

1. **Uses Dockerfile.graalvm** - Builds a native executable ahead-of-time (AOT)
2. **Auto-adjusts defaults**:
   - CPU: 256 units (vs 512 for JVM) - native images need less CPU
   - Memory: 512 MB (vs 2048 for JVM) - 50-70% reduction in memory usage
   - JVM Options: N/A (native images don't use JVM)

3. **Creates folder structure** - Uses `runs/Scenarios/{TYPE}/{scenario}/graalvm/{date}/{run}/` instead of `java25`

4. **Builds and deploys** - Compiles native image (~5-10 min build time) and pushes to ECR like other images

### Why GraalVM Native?

| Aspect | JVM | GraalVM Native |
|--------|-----|----------------|
| **Startup Time** | 3-5 seconds | ~100ms |
| **Memory Usage** | 400-700 MB | 200-300 MB |
| **Image Size** | 250-300 MB | 80-100 MB |
| **Cold Start Performance** | Slower | Much faster |
| **Peak Throughput** | Higher (JIT) | Slightly lower (no JIT) |
| **Build Time** | 1-2 min | 5-10 min |

### Use Cases for GraalVM

- **Serverless/Container spikes** - Instant startup needed
- **Cost optimization** - Lower memory = lower Fargate cost
- **Cold start testing** - Scenario 6 comparisons
- **Edge deployment** - Smaller container images
- **Cost-sensitive production** - Acceptable if throughput difference is <5-10%

### Example: Comparing Java 21 vs GraalVM

```bash
# Run 1: Java 21
npm run setup
# Select: Scenario Type: REST
#         Scenario: Read Heavy
#         Java version: 21
#         CPU: 512, Memory: 2048

# Run 2: GraalVM  
npm run setup
# Select: Scenario Type: REST
#         Scenario: Read Heavy
#         Java version: GraalVM  (CPU/Memory auto-set to 256/512)
```

Both will create identical folder structures except for the version folder name:
- JVM run: `runs/Scenarios/REST/read-heavy/java21/2026-03-02/1/`
- Native run: `runs/Scenarios/REST/read-heavy/graalvm/2026-03-02/1/`

Then deploy both and run identical k6 load tests to compare metrics.

### Example Workflow


```bash
# 1. Create a new run configuration
npm run setup

# 2. (Optional) Bootstrap AWS infrastructure if not already done
npm run aws-bootstrap

# 3. Deploy the generated task to ECS
npm run deploy-task

# 4. Check the service in AWS Console or run benchmarks against it
```

### Benefits

- **No manual ECS console clicks** - Everything automated
- **Consistent deployments** - Same task definition from run config
- **Easy rollback** - Previous task revisions stay in ECS history
- **Container Insights ready** - Automatically enables monitoring on clusters
- **Log group creation** - Creates CloudWatch log groups if they don't exist
